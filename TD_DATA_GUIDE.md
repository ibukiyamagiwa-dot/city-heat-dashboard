# TD（話題変化率）手動入力ガイド

TOKYO CLIMB の主指標 **TD（Topic Delta）** を、SNS API を使わずに毎朝の手動観測で運用するためのガイドです。
入力がある駅は実データの色（🔴🟠🟢）になり、入力がない駅はその日「❓未知ノード」として扱われます。

---

## TD の定義（卒論に書く式）

```text
TD(駅, d) = ( M(駅, d) − M(駅, d−1) ) / max( M(駅, d−1), 1 ) × 100 [%]

M(駅, d) = 日 d の固定観測窓における、クエリ集合 Q(駅) に一致した公開投稿の件数
```

- クエリ集合 Q(駅) は `td_queries.yaml` で定義（変更したら changelog に追記）
- **件数のみ**を記録する。投稿本文・アカウント名は記録しない
- TD の計算には**前日の件数も必要**。運用初日は全駅 ❓ になるのが正常

---

## 毎朝の手順（所要 約15分）

1. **毎日同じ時刻**に観測する（例: 朝 8:00。ズレると前日比の意味が壊れる）
2. X（旧Twitter）の検索を開き、`td_queries.yaml` の **tier A 15駅** のクエリを順に検索
   - 検索タブは「最新」
   - **直近1時間**（例: 7:00〜8:00）に投稿された件数を数える
   - 50件を超えたら数えるのをやめて `50` と記録（上限キャップ）
3. `td_counts.csv` に1駅1行で追記する

```csv
date,station_id,mention_count,note
2026-06-12,shinjuku,34,
2026-06-12,shibuya,41,
2026-06-12,ikebukuro,18,イベント告知が多め
2026-06-12,tokyo,22,
```

- `station_id` は `td_queries.yaml` のキー（ローマ字）を使う
- `note` は空欄でよい（気づいたことがあれば短く）

4. ビルドして push する

## モードの自動判定

`build_tokyo_climb.py` は日付ごとに自動で切り替えます。

| 状態 | その日の扱い |
|------|--------------|
| `td_counts.csv` にその日の行が**1行でもある** | **手動モード**: 入力済み駅は実TD、未入力駅は ❓（td なし） |
| その日の行が**1行もない** | **ダミーモード**: 全駅シード生成のダミーTD（画面に「ダミー」と表示） |

- 手動モードの駅の型分類: TD > +30% → 🔴 ／ +10〜30% → 🟠 ／ −10%未満 → 🟢 ／ それ以外と件数5件未満 → ❓
- 前日の入力がない駅は TD が計算できないため、その日は ❓（件数だけ記録される）

---

## 運用のコツ

- **最初の2日間**は盤面がほぼ ❓ でも正常（前日比が揃うのは3日目から）
- 毎日全駅は不要。**tier A の15駅だけ**続けることが大事
- 観測できなかった日は無理に埋めない（欠損は「霧の濃い日」としてゲーム的に意味を持つ）
- クエリを変えたくなったら `td_queries.yaml` の changelog に日付と理由を書いてから変える

## 卒論用：出典・限界の書き方

```text
[話題性] X（旧Twitter）公開投稿の検索結果を手動集計（固定観測窓・件数のみ・上限50件）。
         クエリ集合と変更履歴は td_queries.yaml に定義。信頼度 low（話題性の代理指標）。
```

限界として明記すること:
- 観測窓が1時間のため、窓外のバズは捉えられない
- 検索結果の網羅性は X の仕様に依存する
- 件数上限50によりトップ駅同士の差は飽和する
