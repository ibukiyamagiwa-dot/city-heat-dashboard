都市熱狂度分析
マルチエージェントシステム

CrewAI による階層型マルチエージェントで、都市のにぎわい・活動しやすさ・注目度を 非専門家にも追える Web ダッシュボード（CITY SONAR 型）を研究・開発する卒業制作です。

制作物: 寄り町 YORIMACHI

スタート地点と使える時間から、今日ふらっと寄れる町を提案する Web アプリ（制作物）。 鉄道データは国土数値情報 N02 由来の rail_graph.json（213駅ノード・MVP路線）。 町マスタは data/towns.json（28エリア）。

試す: yorimachi.html ／ ガイド: YORIMACHI_GUIDE.md

試作: TOKYO CLIMB（Phase 2）

山手線30駅＋分岐7駅＋東西・副都心・半蔵門線（簡略）の48駅グラフ上で、 スタート駅を選ぶと「今日のボス駅」が自動決定され、Slay the Spire 型の分岐マップを 12ホップで登る、1日1ランのプロトタイプです。 駅の色は単一指標 TD（話題変化率: Google Trends 日次関心の前日比%）のみで決まり、 tier A 37駅（山手線30＋主要分岐7）は自動取得・色付き、残り11駅は ❓ です。

遊んでみる: tokyo_climb.html

ビジョン

複数の公開データ（気象・人流・文化施設・現代エンタメ施設など）を統合し、 「今日、その都市はどれだけ活動しやすく、注目されやすいか」を直感的に比較できるようにする。 派手なエンタメアプリではなく、スコアの根拠・信頼度・内訳まで説明可能な研究用ダッシュボードを目指す。

開発プロセス自体も CrewAI の8部門・16エージェントが企画・データ・監査・実装・フィードバックを分担し、 α版→β版の改善ループで反復する構成とする。

現状の問題

  • 施設数の絶対量だけで比較すると大都市（特に東京）が有利になりやすく、卒論としての説得力が弱い。
  • イベント開催数・SNS/検索トレンドなど「その日の熱量」に近いデータは未接続。
  • OSM 由来の現代エンタメ施設は地域の編集密度に依存し、信頼度が公的統計より低い。
  • 駅別ドリルダウンは UI 検証用の試算段階であり、厳密な駅勢圏分析ではない。
  • progress.md は CrewAI 生成文書のため、記述と実装が一致しない場合がある。

デモ: 都市別スコア

下の埋め込みはプロトタイプ画面です。順位・scale_power（絶対規模）・ accessibility_power（人口・人流あたりの充実度）・weather_boost（当日気象）を分けて表示します。

全画面で見る: prototype_app.html ／ 組織図・進捗: team_dashboard.html

課題

  • 絶対量（scale_power）と相対量（accessibility_power）をどう併記し、読み手に誤解なく伝えるか。
  • 文科省・国土数値情報・人口推計と OSM を混ぜず、信頼度ラベルで役割分担を説明し続けること。
  • マルチエージェントの役割分担と、実際のコード変更・データ更新の対応関係を発表で示すこと。
  • プロトタイプは研究用の初期実証であり、地図・時系列・実ユーザーフィードバック収集はこれから。

今後の改善

  • イベント開催数・人流の実データ連携（固定・按分値の置き換え）。
  • Overpass の検索範囲を行政区域ポリゴン化し、駅中心半径での本実装ドリルダウン。
  • 地図表示・時系列（昨日との差・週末との差）の追加。
  • 対象都市の拡大、スコア式 v0.8 以降の検証とデータ品質 KPI の定義。
  • β版 UI と同意確認・出典追跡の強化、研究評価実験の実施。
