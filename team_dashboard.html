データ設計フェーズ

実データ・仮説値・未接続データを分け、卒業研究として説明できる状態にする段階です。

データソース定義

# 卒業研究で使うデータソース定義です。
# trust_level:
#   high   = 実データとして扱える
#   medium = 参考値として扱う
#   low    = 仮説値・固定値。画面上で必ず明示する

weather:
  name: "Open-Meteo 現在気象データ"
  category: "気象"
  acquisition: "公式API"
  trust_level: "high"
  current_usage: "気温・降水量・風速を取得し、都市活動のしやすさの補助指標として使用"
  audit_rule: "取得時刻・出典・API取得失敗時のフォールバック有無を記録する"

culture_venue:
  name: "劇場・音楽堂（社会教育調査）"
  category: "文化施設"
  acquisition: "urban_indicators.csv（文科省 社会教育調査 令和3年度）"
  trust_level: "medium"
  current_usage: "event_venues を culture_venue_bonus に正規化してスコアに使用"
  audit_rule: "施設数は都道府県単位の公的統計。現代エンタメ全体ではなく、文化施設指標として明示する"

modern_entertainment:
  name: "現代エンタメ施設（OSM）"
  category: "現代エンタメ"
  acquisition: "OpenStreetMap / Overpass API（都市中心から半径15kmのカテゴリ別件数）"

スコア設計

# 都市熱狂度スコア設計 v0.7

## 現在のスコア式

都市熱狂度は、都市の絶対的な規模、人口・人流あたりの施設充実度、その日の気象による一時的な追い風・向かい風を分けて評価する。

```text
mania_score = scale_power + accessibility_power + weather_boost
```

```text
scale_power = 35.0 + flow_bonus + culture_venue_bonus + modern_entertainment_bonus
accessibility_power = culture_per_population_bonus + modern_per_population_bonus + culture_per_flow_bonus
weather_boost = temp_score - precip_penalty - wind_penalty
```

## モデル拡張の理由

従来のモデルは、主要駅利用者数や施設数の絶対量を重視していた。そのため、東京のような大都市は人口・人流・施設数の母数が大きく、単純な施設数比較では有利になりやすい。

教授からの指摘である「施設数だけだと東京が勝ちやすい。人口や有効的な人数に対する施設数で見ないと説得力が弱い」を受け、今回のモデルでは次の2つを分けて評価する。

- `scale_power`
  - 人流、文化施設数、現代エンタメ施設数などの絶対量。
  - 「都市としてどれだけ大きな受け皿を持つか」を示す。

- `accessibility_power`
  - 人口あたり・駅利用者あたりの施設充実度。
  - 「都市規模に対して、施設がどれくらい身近に存在するか」を示す。

これにより、「東京が強いのは単に大きいからなのか」「小規模でも人口あたりでは充実している都市があるのか」という問いに答えられる。卒論上は、絶対量と相対量を併記することで、都市規模バイアスを完全に消すのではなく、見える化して検証可能にする狙いがある。

`weather_boost` は従来どおり、気温・降水量・風速など日々変動する要因を別枠で扱う。これにより、構造的な強さと当日の条件を混同しない。

## 構成要素

- `flow_bonus`
  - 出典: 国土交通省 国土数値情報 駅別乗降客数
  - 信頼度: medium
  - 算出: `station_users` を4都市間で正規化し、0.0〜18.0点に変換
  - 意味: 主要駅利用者が多い都市ほど、人流の底力が高い

- `culture_venue_bonus`
  - 出典: 文部科学省 社会教育調査（劇場・音楽堂）
  - 信頼度: medium
  - 算出: `event_venues` を4都市間で正規化し、0.0〜7.0点に変換
  - 意味: 劇場・音楽堂を「現代エンタメ全体」ではなく、公的統計に基づく文化施設指標として扱う

- `modern_entertainment_bonus`
  - 出典: OpenStreetMap / Overpass API
  - 信頼度: low
  - 対象カテゴリ:
    - `amenity=cinema`（映画館）
    - `tourism=museum`（博物館・美術館）
    - `leisure=stadium`（スタジアム）
    - `tourism=theme_park`（テーマパーク）
    - `amenity=arts_centre`（アートセンター）
    - `amenity=nightclub`（クラブ）
    - `amenity=theatre`（劇場）
    - `leisure=sports_centre`（スポーツセンター）
  - 算出: 都市中心から半径15km以内の施設数を4都市間で正規化し、0.0〜8.0点に変換
  - 意味: 文科省の劇場・音楽堂だけでは捉えきれない、現代的な余暇・娯楽施設の厚みを補助的に評価する
  - 表示: 合計値だけでなく `modern_entertainment_breakdown` としてカテゴリ別件数も画面に表示する

- `accessibility_power`
  - 出典: 総務省人口推計、国土数値情報駅利用者数、文科省社会教育調査、OSM
  - 信頼度: medium〜low
  - 使用指標:
    - `culture_venues_per_100k` = 文化施設数 / 人口 * 100,000
    - `modern_entertainment_per_100k` = 現代エンタメ施設数 / 人口 * 100,000
    - `culture_venues_per_million_station_users` = 文化施設数 / 駅利用者数 * 1,000,000
  - 意味: 絶対数だけではなく、都市規模に対して施設がどれだけ充実しているかを見る

- `temp_score`
  - 出典: Open-Meteo
  - 信頼度: high
  - 算出: 気温を0.0〜18.0点に変換
  - 意味: 気温が活動しやすいほど今日のブーストが上がる

- `precip_penalty` / `wind_penalty`
  - 出典: Open-Meteo
  - 信頼度: high
  - 意味: 降水・強風は屋外活動を下げる可能性があるため減点

## UI上の説明

画面では、総合スコアだけでなく以下を表示する。

- 順位
- `scale_power`（絶対量の規模パワー）
- `accessibility_power`（人口・人流あたりの充実度）
- `weather_boost`（今日のブースト）
- 強み（例: 主要駅利用者数が4都市中1位）
- 弱み（例: 人口あたり文化施設数が4都市中最少）

これにより、点数差から「なぜ高いのか」「なぜ低いのか」に加えて、「大都市だから強いだけなのか」「人口あたりではどうなのか」という問いを自然に確認できるようにする。

加えて、現代エンタメ施設についてはカテゴリ別内訳を表示する。合計値だけでは「何をエンタメとして数えたのか」が見えないため、映画館・博物館・スタジアム・テーマパークなどの内訳を見せることで、教授や非エンジニアがスコア根拠を追跡しやすくする。

## 都市内ドリルダウン / 駅別ビュー

都市単位の比較は卒論上の主スコアとして維持する。ただし、東京・大阪のような大都市を一括りにすると、新宿、渋谷、東京駅、梅田、難波のような都市内部の違いが見えにくい。

そこで次の拡張として、都市詳細の中に「注目駅」ビューを追加する。これは本体の `mania_score` 式を変更するものではなく、都市スコアを説明した後に、その都市の内部を駅単位で掘るためのプロトタイプ枠である。

駅別ビューでは以下を表示する。

- 乗降客数
- 周辺OSM現代エンタメ施設数のプロトタイプ推定
- 文化/エンタメ密度（駅利用100万人あたりの施設数）
- 仮の駅別熱狂度
- 「人流型」「カルチャー型」「総合熱狂型」「バランス型」の分類

分類基準は次の通り。

- `人流型`
  - 駅利用者数が都市内1位、かつエンタメ密度順位が3位以下。
  - 人の流れそのものが駅別熱狂度を押し上げるタイプ。

- `カルチャー型`
  - エンタメ密度順位が都市内2位以内、かつ駅利用者数順位が3位以下。
  - 規模よりも、駅利用者あたりの文化・エンタメ施設の濃さが強いタイプ。

- `総合熱狂型`
  - 駅利用者数順位が2位以内、かつエンタメ密度順位も2位以内。
  - 人流と施設密度の両方が強いタイプ。

- `バランス型`
  - 上記3タイプに当てはまらない駅。
  - 特定の軸に偏りすぎず、複数要素で平均的に支えるタイプ。

現時点の駅周辺OSM現代エンタメ施設数は、都市OSM合計を駅利用者比で按分した試算である。これは「都市→駅へ掘れるUI構造」を検証するためのものであり、厳密な駅勢圏分析ではない。今後は駅中心半径500m〜1kmでOSMを直接取得し、駅勢圏人口や昼間人口を使って検証精度を高める。

## 公的統計とOSMの信頼度の違い

公的統計（国土数値情報、文科省社会教育調査、総務省人口推計）は、調査方法や基準年が明示されており、卒論で根拠として説明しやすい。一方で、都道府県単位である、調査年が少し古い、開催数ではなく施設数であるといった限界がある。

OpenStreetMapは、映画館・テーマパーク・クラブなど現代エンタメに近い施設種別を柔軟に取得できる利点がある。しかし、登録状況が地域ごとの編集密度に依存し、タグ表記にも揺れがある。また、同じ施設が `node`、`way`、`relation` の複数形式で登録されている場合や、複数タグを持つ場合には重複カウントの可能性がある。そのため、本研究ではOSMを高信頼の公的統計と同列には扱わず、`modern_entertainment_bonus` として低信頼の補助指標に位置づける。

この設計により、「信頼できる公的統計で土台を作り、OSMで現代エンタメの不足部分を補う」という役割分担を明確にする。

## 廃止したもの

- 固定 `event_bonus=12.0`（仮説値）→ 廃止
- `event_bonus` 合成フィールド → 廃止（底力と気象を分離）
- `venue_bonus` という曖昧な名称 → `culture_venue_bonus` に再定義
